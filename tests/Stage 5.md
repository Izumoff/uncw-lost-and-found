# Sprint 3

## User Story: Campus Community User can edit own Found Item report before approval

| Test Case | Input | Expected Output | Actual Output |
|---|---|---|---|
| 1 (Success) | **Precondition:** User is logged in as the owner of an unapproved Found report. 1. Open `/reports/<report_id>/edit/found/` 2. View the form | The edit page opens and the report fields are available for editing. | The edit page opened successfully for the correct owner account. |
| 2 (Failure  Fixed) | **Precondition:** Report was created by a different user account. 1. Log in as another user 2. Open `/reports/4/edit/found/` | The system blocks access because the current user is not the owner. | Initial result: redirected to Published Reports list. Investigation showed the report belonged to another user. Diagnostic message displayed: **ōBlocked: report owner does not match current user.ö** **Fixed:** confirmed this was expected behavior for non-owner access. |
| 3 (Success) | **Precondition:** User is logged in as owner of an unapproved Found report. 1. Open `/reports/<report_id>/edit/found/` 2. Verify existing values in fields | Existing report data is prefilled in the form. | Form opened with existing report data for the correct owner account. |
| 4 (Failure  Fixed) | **Precondition:** User is logged in as owner of an unapproved Found report. 1. Open edit page 2. Modify fields 3. Click **Save Changes** | The report is updated and the user receives clear visual feedback that changes were saved. | Initial result: changes saved, but there was **no clear visual feedback** after submit. **Fixed:** redirect now returns to `/reports/<report_id>/edit/found/?updated=1` and displays success message: **ōYour report has been updated successfully.ö** |
| 5 (Success) | **Precondition:** User is on edit page for own unapproved Found report. 1. Check page heading 2. Check submit button text | Edit mode clearly shows edit-specific UI text. | Edit page displays **ōEdit Found Item Reportö** and button text **ōSave Changes.ö** |
| 6 (Success) | **Precondition:** User is on create page `/reports/create/found/`. 1. Open create page 2. Check heading and button | Create mode remains unchanged. | Create page still works correctly with heading **ōCreate Found Item Reportö** and button **ōSubmit Report.ö** |
| 7 (Failure  Fixed) | **Precondition:** User owns an unapproved Found report. 1. Open `/reports/<report_id>/` for an unapproved report | Owner should be able to open the detail page for own unapproved report. | Initial result: not possible because `report_detail` allowed only published reports. **Fixed:** owner can now open own unapproved report detail page successfully. |
| 8 (Success) | **Precondition:** User is viewing detail page for own unapproved Found report. 1. Open `/reports/<report_id>/` 2. Check available actions | **Edit Report** button is visible only when editing is allowed. | **Edit Report** button is visible for owner on own unapproved Found report. |
| 9 (Success) | **Precondition:** User is viewing detail page for report not eligible for owner edit action. 1. Open detail page as non-owner, or open published Found report, or open non-Found report | **Edit Report** button is not shown. | Button is hidden when editing is not allowed. |
| 10 (Success) | **Precondition:** Existing Stage 3 and Stage 4 flows are available. 1. Open reports list 2. Open report detail 3. Open create Found report page | Existing flows continue to work without regression. | Existing routes and flows continued to work during Stage 5 implementation. |
